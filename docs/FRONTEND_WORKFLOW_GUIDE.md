# 프론트엔드 개발 및 백엔드 연동 워크플로우 가이드

이 문서는 효율적인 프론트엔드 개발과 백엔드 팀과의 원활한 협업을 위한 표준 작업 절차를 정의합니다.

## 1. UI/UX 개발 및 목업(Mock) 테스트 단계
**목표:** 사용자 인터페이스의 동작과 흐름을 우선적으로 확정하고 검증합니다.

*   **UI 컴포넌트 및 페이지 구현**: 기획안 또는 시나리오에 맞춰 화면을 구현합니다.
*   **Mocking 적용**: 백엔드 API가 준비되지 않았거나 연동 전인 경우, 프론트엔드 내부에서 가짜 데이터(Mock Data)와 가짜 함수(Mock Function)를 사용하여 로직을 구현합니다.
    *   `setTimeout` 등을 활용하여 비동기 통신 지연을 시뮬레이션합니다.
    *   성공(Success) 및 실패(Error) 케이스를 강제로 발생시켜 UI 반응을 점검합니다.
*   **UI/UX 검증**: 화면 흐름, 입력 폼 유효성 검사, 모달 동작, 에러 메시지 표시 등이 기획 의도대로 동작하는지 확인합니다.

## 2. 인터페이스 규격 문서화 (Frontend Spec)
**목표:** UI에서 실제로 필요한 데이터와 동작을 정리하여 백엔드 개발 기준을 마련합니다.

*   **데이터 구조 정의**: UI를 구성하기 위해 필요한 데이터 필드(Request Body, Response Data)를 정리합니다.
*   **API 요구사항 도출**: 필요한 API 엔드포인트, HTTP 메서드, 파라미터 등을 명세합니다.
*   **시나리오 정리**: "사용자가 A를 입력하고 B 버튼을 누르면 C 데이터가 전송되어야 함"과 같은 구체적인 동작 시나리오를 문서화합니다.
    *   *예시: `docs/account/signup/file-upload-admin-approval/프론트엔드_시나리오.md`*

## 3. 백엔드 교차 검증 (Cross-Verification)
**목표:** 프론트엔드 요구사항이 백엔드 로직 및 데이터베이스 구조와 일치하는지 확인합니다.

*   **문서 공유**: 작성된 인터페이스 규격 문서를 백엔드 팀(또는 담당자)에게 전달합니다.
*   **피드백 및 조율**:
    *   백엔드 관점에서 구현 불가능하거나 비효율적인 부분이 있는지 검토합니다.
    *   보안 이슈, 데이터 무결성 문제 등을 논의하여 API 스펙을 확정합니다.
*   **최종 스펙 합의**: 양측이 합의한 최종 API 명세서를 확정합니다.

## 4. API 연동 (Integration)
**목표:** Mock 로직을 제거하고 실제 백엔드 API와 연결합니다.

*   **Service Layer 구현**: `axios` 등을 사용하여 실제 API를 호출하는 서비스 함수를 작성합니다.
*   **Mock 제거 및 교체**: 컴포넌트 내의 테스트용 Mock 코드를 주석 처리하거나 제거하고, 실제 서비스 함수로 교체합니다.
*   **예외 처리 강화**: 실제 네트워크 에러, 서버 내부 에러(500), 인증 에러(401/403) 등에 대한 구체적인 처리를 추가합니다.

## 5. 통합 테스트 (Final Testing)
**목표:** 실제 데이터 흐름을 포함한 전체 기능을 검증합니다.

*   **E2E 테스트**: 사용자의 진입부터 최종 완료까지의 흐름이 끊김 없이 동작하는지 확인합니다.
*   **데이터 검증**: 프론트엔드에서 보낸 데이터가 DB에 정확히 저장되는지, 조회 시 올바르게 표시되는지 확인합니다.
*   **버그 수정**: 연동 과정에서 발견된 이슈를 수정하고 재배포합니다.
